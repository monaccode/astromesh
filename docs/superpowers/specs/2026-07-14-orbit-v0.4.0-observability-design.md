# Astromesh Orbit v0.4.0 — Observability (GCP) — Design

**Date:** 2026-07-14
**Status:** Approved (design)
**Scope:** `astromesh-orbit` (GCP provider only)
**Depends on:** the core OTLP export wiring (spec `2026-07-14-otlp-export-wiring-design.md`), now shipped.

---

## Summary

Deliver the observability content that Orbit's roadmap promised under "v0.2.0" but never
implemented. Four deliverables:

1. **Cloud Monitoring dashboard** — a `google_monitoring_dashboard` with Cloud Run's golden
   signals. On by default (it is a single Terraform resource; it costs nothing).
2. **Cloud Trace via an OpenTelemetry Collector sidecar** — opt-in. The sidecar receives OTLP
   on `localhost:4317` and exports to Cloud Trace. The runtime container is configured to
   actually emit (`ASTROMESH_OTLP_ENABLED=1`).
3. **`orbit logs`** — read the deployment's Cloud Run logs from Cloud Logging.
4. **`orbit upgrade`** — re-render the Terraform templates after an Orbit package update and
   show a diff against what is currently generated.

Shipped as **v0.4.0**, with the roadmap renumbered (see Versioning).

## Background & Constraints

### The v0.2.0 gap

Orbit's roadmap lists a "v0.2.0 — Observability" release (Cloud Monitoring, Cloud Trace, Cloud
Logging, dashboard, `orbit logs`, `orbit upgrade`). The package **was** published as 0.2.0, but
**none of that content exists**: there is no monitoring template (the 12 templates are
main/variables/backend/iam/networking/cloud_sql/memorystore/secrets/cloud_run/storage/
artifact_registry/outputs), no Cloud Monitoring/Trace/Logging code anywhere in the package, and
no `orbit logs` or `orbit upgrade` commands. The version number was consumed without its
content. This spec delivers it.

### The core dependency (why this only works now)

Astromesh's OTLP export path existed but was never wired: nothing constructed
`TelemetryManager`/`MetricsManager`/`OTLPCollector`, so the application exported **nothing**. A
collector sidecar would have received an empty stream. The core wiring spec fixed that: with
`ASTROMESH_OTLP_ENABLED=1` the runtime now starts a `TelemetryManager` and installs an
`OTLPCollector` at bootstrap, exporting OTLP spans to `OTEL_EXPORTER_OTLP_ENDPOINT` (default
`http://localhost:4317`).

That default is exactly where a Cloud Run sidecar listens — so Orbit needs **no core changes**,
only to run the sidecar and set two environment variables on the runtime container.

### What Cloud Run already gives for free

- **Logs:** Cloud Run ships each container's stdout/stderr to Cloud Logging automatically. Orbit
  provisions nothing for this; `orbit logs` simply reads them.
- **Metrics:** Cloud Run request count, latency, instance count, and CPU/memory utilization are
  collected by Cloud Monitoring automatically. The dashboard only *displays* them.

There is therefore **no `logging` config toggle**: it would have nothing to configure.
Astromesh's application logs are plain text (`logging_config.py` installs a
`logging.Formatter`, not a JSON one), so they land in Cloud Logging as unstructured text lines.
Making them *structured* (JSON with severity and fields) requires a JSON formatter in the
astromesh core — **out of scope, tracked as a follow-up**.

## Config Schema

New `spec.observability` section. All fields defaulted → **backward compatible** (existing
`orbit.yaml` files remain valid without it).

```yaml
spec:
  # ... provider / compute / database / cache / secrets / images / storage unchanged ...
  observability:
    dashboard: true                # Cloud Monitoring dashboard (free; on by default)
    tracing:
      enabled: false               # OTel Collector sidecar -> Cloud Trace (adds a container)
      collector_image: "otel/opentelemetry-collector-contrib:0.115.1"
```

`tracing` defaults to **off**: the sidecar adds a container (and its CPU/memory) to every Cloud
Run instance, so it is an explicit opt-in. The dashboard defaults to **on** because it is a
single Terraform resource over metrics that already exist.

`collector_image` is configurable rather than hardcoded — see Risks.

### `config.py` (Pydantic models)

```python
class TracingSpec(BaseModel):
    enabled: bool = False
    collector_image: str = "otel/opentelemetry-collector-contrib:0.115.1"

class ObservabilitySpec(BaseModel):
    dashboard: bool = True
    tracing: TracingSpec = TracingSpec()

class OrbitSpec(BaseModel):
    # ... existing fields ...
    observability: ObservabilitySpec = ObservabilitySpec()
```

## Terraform

All new resources render **only** when their flag is true — `enabled: false` produces no
resource, no env var, no IAM binding, and no dangling reference.

### New: `monitoring.tf.j2`

A `google_monitoring_dashboard` (guarded by `observability.dashboard`) whose `dashboard_json`
tiles chart Cloud Run's golden signals for the runtime service:

- Request count (`run.googleapis.com/request_count`)
- Request latency, p50 and p95 (`run.googleapis.com/request_latencies`)
- Instance count (`run.googleapis.com/container/instance_count`)
- CPU and memory utilization (`run.googleapis.com/container/cpu/utilizations`,
  `.../memory/utilizations`)

### Modified: `cloud_run.tf.j2`

Cloud Run v2 supports multiple containers. When `observability.tracing.enabled`:

1. The runtime container gains `name = "runtime"` and
   `depends_on = ["otel-collector"]` (Cloud Run starts the sidecar first).
2. Two env vars are added to the runtime container — **this is what makes the core actually
   export**:
   - `ASTROMESH_OTLP_ENABLED = "1"`
   - `OTEL_EXPORTER_OTLP_ENDPOINT = "http://localhost:4317"`
3. A second container is added — the collector sidecar. It declares **no `ports`** (only the
   ingress container does) and takes its config inline from an environment variable:

```hcl
    containers {
      name  = "otel-collector"
      image = "{{ observability.tracing.collector_image }}"
      args  = ["--config=env:OTEL_COLLECTOR_CONFIG"]

      env {
        name  = "OTEL_COLLECTOR_CONFIG"
        value = <<-EOT
          receivers:
            otlp:
              protocols:
                grpc:
                  endpoint: 0.0.0.0:4317
          exporters:
            googlecloud:
              project: ${var.project_id}
          service:
            pipelines:
              traces:
                receivers: [otlp]
                exporters: [googlecloud]
        EOT
      }

      resources {
        limits = { cpu = "1", memory = "512Mi" }
      }
    }
```

The collector authenticates to Cloud Trace with the Cloud Run service account — no keys.

### Modified: `iam.tf.j2`

When tracing is enabled, grant the `orbit` service account:
- `roles/cloudtrace.agent` (write spans)
- `roles/monitoring.metricWriter` (write metrics)

### Modified: `validators.py`

Add to `REQUIRED_APIS`: `monitoring.googleapis.com`, `cloudtrace.googleapis.com`,
`logging.googleapis.com`.

### Modified: `provider.py`

Register `monitoring.tf.j2` in `TEMPLATE_FILES` and add `"observability": config.spec.observability`
to `_build_context`. (`monitoring.tf.j2` must be registered before `cloud_run.tf.j2`/`iam.tf.j2`
are taught to read `observability`, so `generate()` never renders a template with an undefined
variable — the same ordering discipline used for `storage` in v0.3.0.)

### Modified: `wizard/defaults.py`

Both presets gain an `observability` block (`dashboard: true`, `tracing.enabled: false`).

## CLI

### `orbit logs`

Reads the runtime service's logs from Cloud Logging by shelling out to `gcloud` — reusing the
existing async `_run_gcloud()` helper pattern in `providers/gcp/validators.py`, which already
handles Windows's `gcloud.cmd`. **No new dependency.**

```
astromeshctl orbit logs [--limit 50] [--since 1h] [--config orbit.yaml]
```

Runs, in effect:

```
gcloud logging read
  'resource.type=cloud_run_revision AND resource.labels.service_name=astromesh-runtime'
  --project=<project> --limit=<limit> --freshness=<since> --format=json
```

and renders the entries (timestamp, severity, message) as a `rich` table, consistent with
`orbit status`.

### `orbit upgrade`

Purely local; **no cloud calls**. Re-renders the provider's templates from the current
`orbit.yaml` into a temporary directory, then produces a unified diff (`difflib.unified_diff`)
against each corresponding file in `.orbit/generated/`. Prints the diff; with `--apply`, writes
the newly-rendered files over the generated ones.

```
astromeshctl orbit upgrade [--config orbit.yaml] [--apply]
```

Purpose: after upgrading the `astromesh-orbit` package, show exactly what the new templates
would change before any `terraform apply`.

## Data Flow (tracing enabled)

```
Agent run → TracingContext spans → OTLPCollector (installed by the core at bootstrap,
  because ASTROMESH_OTLP_ENABLED=1)
    → OTLP/gRPC to OTEL_EXPORTER_OTLP_ENDPOINT (http://localhost:4317)
      → otel-collector sidecar (same Cloud Run instance)
        → googlecloud exporter (authenticated as the orbit service account)
          → Cloud Trace
```

Metrics and logs need no pipeline: Cloud Run reports them to Cloud Monitoring / Cloud Logging
directly, and the dashboard and `orbit logs` read them.

## Error Handling

| Scenario | Behavior |
|---|---|
| `monitoring` / `cloudtrace` / `logging` API not enabled | `validate()` fails before Terraform with a `gcloud services enable ...` remediation. |
| `tracing.enabled: false` | No sidecar container, no IAM bindings, no OTLP env vars. The runtime's export stays off (the core defaults to off), so nothing is emitted into a void. |
| `dashboard: false` | No dashboard resource, no output. |
| `orbit logs` without credentials / gcloud | The `_run_gcloud` helper's non-zero exit is surfaced as a clear error with the `gcloud auth login` remediation, matching `validate()`'s existing behavior. |
| Collector sidecar fails to start | Cloud Run surfaces it; the runtime container still serves (spans are dropped by the OTel SDK's exporter, which never blocks a request). |

## Testing (offline — no live GCP calls, matching the existing suite)

- **Config:** `observability` defaults (`dashboard is True`, `tracing.enabled is False`,
  `collector_image` non-empty); an `orbit.yaml` with no `observability` block still parses.
- **Templates:**
  - `monitoring.tf` renders the dashboard; with `dashboard: false` it is absent.
  - `cloud_run.tf` with `tracing.enabled: true` contains the `otel-collector` container, the
    `depends_on`, `ASTROMESH_OTLP_ENABLED`, and `OTEL_EXPORTER_OTLP_ENDPOINT`; with tracing
    disabled **none** of those appear.
  - `iam.tf` contains `roles/cloudtrace.agent` only when tracing is enabled.
- **Provider:** `generate()` writes `monitoring.tf`.
- **Validators:** the three new APIs are in `REQUIRED_APIS`.
- **Wizard:** both presets carry an `observability` block.
- **CLI:** `orbit logs` builds the expected `gcloud logging read` argument list (the subprocess
  helper is mocked — no real gcloud call); `orbit upgrade` produces a diff when a generated file
  differs from a freshly-rendered one, and an empty diff when they match (tmp dirs).

**Gate:** the full `astromesh-orbit` suite green, and ruff clean (the `test-orbit` CI job now
runs lint + format + tests).

## Risks (declared up front)

**The collector image tag and its config mechanism must be verified at implementation time.**
This design assumes the `otel/opentelemetry-collector-contrib` image supports
`--config=env:VAR` (the collector's `env:` config provider) and ships the `googlecloud`
exporter. Both are believed true, but were not verified against a live registry while writing
this spec. Mitigation: `collector_image` is a **config field with a pinned default**, not a
hardcoded value, so a wrong tag is a one-line fix rather than a template rewrite. The
implementer must confirm the image, tag, and `--config=env:` support before finalizing the
sidecar template, and adjust the default if needed.

## Component Boundaries

- **`config.py`** — parses and validates `observability`; interface = typed Pydantic models; no
  GCP knowledge.
- **`monitoring.tf.j2`** — renders the dashboard from the `observability` context var only.
- **`cloud_run.tf.j2`** — owns the container spec, including the sidecar and the two OTLP env
  vars; depends on the `orbit` service account from `iam.tf.j2`.
- **`cli.py` → `logs`** — builds a `gcloud logging read` invocation and renders the result;
  depends on the gcloud subprocess helper, not on Terraform.
- **`cli.py` → `upgrade`** — renders templates to a temp dir and diffs; depends on the provider's
  `generate()` and `difflib` only. No cloud, no Terraform.
