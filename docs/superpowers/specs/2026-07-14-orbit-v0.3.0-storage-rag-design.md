# Astromesh Orbit v0.3.0 — Storage & RAG (GCP) — Design

**Date:** 2026-07-14
**Status:** Approved (design)
**Scope:** `astromesh-orbit` (GCP provider only)
**Supersedes roadmap item:** "Cloud CDN for Studio" (dropped — see Background)

---

## Summary

Advance Astromesh Orbit from v0.2.0 (Observability) to **v0.3.0 (Storage & RAG)** for
the GCP provider. Deliver the storage substrate that Astromesh's RAG stack can actually
use today:

1. **GCS bucket for RAG source documents** — provisioned, IAM-wired to the runtime
   service account, and exposed to the runtime via `ASTROMESH_RAG_BUCKET`.
2. **Artifact Registry (Docker)** — a repository for custom Astromesh container images,
   with pull access granted to the runtime service account.
3. **pgvector on the existing Cloud SQL** — the RAG vector store. **No new managed vector
   database is provisioned.** Astromesh's `PGVectorStore` enables the extension on first
   use and Cloud SQL for PostgreSQL supports it natively, so the vector store is the
   Cloud SQL instance Orbit already deploys.

A secondary deliverable is fixing the **pre-existing red test suite** in `astromesh-orbit`
(drift left by commit `6278ccc`).

---

## Background & Constraints

### Astromesh RAG base (what actually exists)

- `astromesh/rag/` ships vector stores `faiss`, `chroma`, `pgvector`, `qdrant`
  (`astromesh/rag/stores/`) and builds pipelines from `*.rag.yaml` specs
  (`astromesh/rag/loader.py`, `factory.py`).
- Documents are ingested as **raw strings** via `POST /v1/rag/ingest`
  (`astromesh/api/routes/rag.py` → `RAGPipeline.ingest(document, metadata, ...)`).
  There is **no object-storage document loader** in the core (`grep` for
  `google.cloud.storage` / `gcsfs` / `boto3` → none).
- `PGVectorStore` (`astromesh/rag/stores/pgvector.py`) connects via a plain
  `postgresql://` DSN and, on first pool acquisition, runs
  `CREATE EXTENSION IF NOT EXISTS vector` and creates its embeddings table. Cloud SQL for
  PostgreSQL supports the `vector` extension via `CREATE EXTENSION vector` (no instance
  flag required).

**Design consequence:** the vector store on GCP is the existing Cloud SQL — Orbit must
not add a redundant vector DB. The GCS RAG bucket is a **document staging substrate**:
Orbit provisions it and wires the env var + IAM, but the runtime cannot read from it until
the core adds a GCS loader (explicitly out of scope, tracked as future core work).

### Studio was deliberately removed

Commit `6278ccc` ("remove astromesh-cloud and studio from docs and orbit deployment")
states: *"Cloud and Studio are no longer deployed."* It removed Studio/cloud-api from the
Orbit config schema, provider, templates, wizard, and deleted the `cloud/` docs pages.
Today Orbit deploys only the `runtime` Cloud Run service.

The v0.3.0 roadmap item **"Cloud CDN for Studio"** therefore targets a component that was
intentionally removed. **Decision (approved): drop it.** Re-introducing Studio would
reverse `6278ccc`. `roadmap.md` is updated to reflect this.

### Pre-existing red test suite

The `astromesh-orbit` suite is currently **10 failed / 21 errors**. Root causes, both left
over from `6278ccc`:

- `tests/conftest.py` `SAMPLE_ORBIT_YAML` uses `"environment": "dev"`, but
  `OrbitMetadata.environment` is `Literal["develop", "staging", "production"]` → every test
  that builds a config from the fixture raises `ValidationError` (the 21 errors).
- `tests/test_config.py::test_parse_compute_specs` still asserts
  `config.spec.compute.studio.min_instances == 0`, but `ComputeSpec` only has `runtime`.

These are fixed as part of this change (targeted cleanup in the area we are working in).

## Config Schema

New `spec.storage` section, all fields defaulted → **backward compatible** (existing
`orbit.yaml` files remain valid without a `storage` block).

```yaml
apiVersion: astromesh/v1
kind: OrbitDeployment
metadata:
  name: my-astromesh
  environment: develop
spec:
  provider: { name: gcp, project: my-project, region: us-central1 }
  # ... compute / database / cache / secrets / images unchanged ...
  storage:
    rag_documents:
      enabled: true
      versioning: true
    artifact_registry:
      enabled: true
      repository: ""      # empty → defaults to "<metadata.name>-images"
```

### `config.py` (Pydantic models)

```python
class RagDocumentsSpec(BaseModel):
    enabled: bool = True
    versioning: bool = True

class ArtifactRegistrySpec(BaseModel):
    enabled: bool = True
    repository: str = ""   # resolved to "<name>-images" at render time if empty

class StorageSpec(BaseModel):
    rag_documents: RagDocumentsSpec = RagDocumentsSpec()
    artifact_registry: ArtifactRegistrySpec = ArtifactRegistrySpec()

class OrbitSpec(BaseModel):
    # ... existing fields ...
    storage: StorageSpec = StorageSpec()
```

### `core/resources.py`

Mirror `StorageSpec` / `RagDocumentsSpec` / `ArtifactRegistrySpec` as frozen dataclasses to
keep parity with the existing `ComputeSpec`/`DatabaseSpec`/etc. (as `6278ccc` maintained
both in sync).

## Terraform Templates

All new resources render **only when their `enabled` flag is true**, so `enabled: false`
produces no resource and no dangling reference.

### New: `storage.tf.j2`

```hcl
{% if storage.rag_documents.enabled %}
resource "google_storage_bucket" "rag_documents" {
  name                        = "${var.project_id}-{{ meta.name }}-rag-docs"
  location                    = var.region
  project                     = var.project_id
  force_destroy               = false
  uniform_bucket_level_access = true

  versioning {
    enabled = {{ 'true' if storage.rag_documents.versioning else 'false' }}
  }
}

resource "google_storage_bucket_iam_member" "rag_documents_runtime" {
  bucket = google_storage_bucket.rag_documents.name
  role   = "roles/storage.objectAdmin"
  member = "serviceAccount:${google_service_account.orbit.email}"
}
{% endif %}
```

Bucket name uses the `{project_id}-` prefix; because GCP project IDs are globally unique,
this avoids the cross-project name collisions the state bucket handles with a hash suffix.

### New: `artifact_registry.tf.j2`

```hcl
{% if storage.artifact_registry.enabled %}
resource "google_artifact_registry_repository" "images" {
  location      = var.region
  repository_id = "{{ storage.artifact_registry.repository or (meta.name ~ '-images') }}"
  format        = "DOCKER"
  project       = var.project_id
  description   = "Custom Astromesh container images"
}

resource "google_artifact_registry_repository_iam_member" "images_runtime_reader" {
  location   = var.region
  repository = google_artifact_registry_repository.images.repository_id
  role       = "roles/artifactregistry.reader"
  member     = "serviceAccount:${google_service_account.orbit.email}"
  project    = var.project_id
}
{% endif %}
```

### Modified: `cloud_run.tf.j2`

Add, inside the runtime container's env block:

```hcl
{% if storage.rag_documents.enabled %}
      env {
        name  = "ASTROMESH_RAG_BUCKET"
        value = google_storage_bucket.rag_documents.name
      }
{% endif %}
```

### Modified: `outputs.tf.j2`

```hcl
{% if storage.rag_documents.enabled %}
output "rag_bucket" {
  value       = google_storage_bucket.rag_documents.name
  description = "GCS bucket for RAG source documents"
}
{% endif %}
{% if storage.artifact_registry.enabled %}
output "artifact_registry_repo" {
  value       = "${var.region}-docker.pkg.dev/${var.project_id}/${google_artifact_registry_repository.images.repository_id}"
  description = "Artifact Registry Docker repository path"
}
{% endif %}
```

## Provider, Validators, Wizard

### `providers/gcp/provider.py`

- Add `"storage.tf.j2"` and `"artifact_registry.tf.j2"` to `TEMPLATE_FILES` (before
  `outputs.tf.j2`; Terraform resolves refs regardless of file order).
- Add `"storage": config.spec.storage` to `_build_context()`.
- `provision()` already dumps all Terraform outputs to `orbit.env`, so `rag_bucket` /
  `artifact_registry_repo` flow through automatically. No `status()` change required
  (keeps status focused on the runtime service).

### `providers/gcp/validators.py`

Add to `REQUIRED_APIS`:

```python
"storage.googleapis.com",
"artifactregistry.googleapis.com",
```

Both are enabled unconditionally (cheap to enable; both features default on). A missing API
produces the existing check failure with a `gcloud services enable ...` remediation.

### `wizard/defaults.py`

Add a `storage` block to both presets (starter and pro) and include it in
`build_orbit_yaml()`:

```python
"storage": {
    "rag_documents": {"enabled": True, "versioning": True},
    "artifact_registry": {"enabled": True, "repository": ""},
},
```

GCS + Artifact Registry are negligible at rest, so `estimated_cost` is unchanged
(15 / 80). A one-line note is added to docs.

## pgvector / RAG Wiring (documentation, no new resource)

Orbit provisions **no** vector database. The RAG vector store on GCP is the existing Cloud
SQL. Operators enable it per RAG pipeline with a `*.rag.yaml`:

```yaml
kind: RAGPipeline
metadata: { name: docs }
spec:
  embeddings: { provider: sentence_transformers, model: all-MiniLM-L6-v2, dimension: 384 }
  vector_store:
    backend: pgvector
    collection: embeddings
    connection:                # or a single `dsn:` value
      host: /cloudsql/<CONNECTION_NAME>   # Cloud SQL socket the runtime already mounts
      user: astromesh
      password: <from secret>
      database: astromesh
```

The runtime already receives `ASTROMESH_DATABASE_URL` and mounts the Cloud SQL socket at
`/cloudsql`, so no new connectivity is needed. Documenting this snippet is the deliverable;
`PGVectorStore` creates the extension and table on first use.

**Out of scope (future core work):** a GCS document loader in `astromesh/rag/` that reads
source files from `ASTROMESH_RAG_BUCKET`, and defaulting the RAG pipeline DSN from the
environment. These belong to the `astromesh` core repo, not Orbit.

## Data Flow

`orbit apply` →
`validate()` (checks include `storage.googleapis.com`, `artifactregistry.googleapis.com`) →
`generate()` (renders `storage.tf`, `artifact_registry.tf`, updated `cloud_run.tf` /
`outputs.tf`) →
`terraform init/apply` → GCS bucket + Artifact Registry created; runtime SA granted
`storage.objectAdmin` on the bucket and `artifactregistry.reader` on the repo; runtime gets
`ASTROMESH_RAG_BUCKET` →
outputs `rag_bucket` and `artifact_registry_repo` written to `orbit.env`.

`orbit eject` includes the new templates. `orbit destroy` removes them; because the bucket
is `force_destroy = false`, a non-empty bucket must be emptied first — documented in the
error-handling / destroy notes.

---

## Error Handling

| Scenario | Behavior |
|---|---|
| `storage` / `artifactregistry` API not enabled | `validate()` fails before Terraform with a `gcloud services enable` remediation. |
| Bucket name collision | Prevented in practice by the globally-unique `{project_id}-` prefix. |
| `destroy` on a non-empty RAG bucket | `force_destroy = false` → Terraform refuses; docs instruct emptying the bucket first (documents are user data — deliberately not auto-deleted). |
| `enabled: false` | No resource, no output, no env var, no dangling reference. |

## Component Boundaries

Each unit has one purpose, a clear interface, and is independently testable:

- **`config.py` / `resources.py`** — parse & validate `storage`; interface = typed models;
  depends on nothing GCP.
- **`storage.tf.j2` / `artifact_registry.tf.j2`** — render HCL from the `storage` context;
  interface = Jinja context vars; depends on the `orbit` SA resource from `iam.tf.j2`.
- **`provider.py`** — orchestrates render order + context; interface = `OrbitProvider`
  Protocol; depends on templates + config.
- **`validators.py`** — API enablement checks; interface = `list[CheckResult]`.
- **`wizard/defaults.py`** — preset → `orbit.yaml` dict; interface = `build_orbit_yaml()`.
